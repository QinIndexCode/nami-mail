# 安全策略

[简体中文](SECURITY.md) | [English](SECURITY.en.md)

Nami Mail 处理账户凭据、OAuth 刷新令牌和本地邮件缓存。请不要把安全问题、真实邮件、令牌、附件或可用于定位用户的数据发布到公开 Issue、讨论区或 Pull Request。

## 支持范围

| 版本 | 状态 |
| --- | --- |
| 当前 `0.1.x` 主线 | 接受安全报告并按影响评估修复 |
| 旧版本或自行修改的构建 | 不承诺修复；请先升级或在当前主线复现 |

项目尚未承诺固定修复时限。报告确认后，维护者会在不泄露其他用户数据的前提下说明处理进展。

## 报告方式

请使用 [GitHub 私密漏洞通报（Report a vulnerability）](https://github.com/QinIndexCode/nami-mail/security/advisories/new) 创建安全报告。该仓库已启用 private vulnerability reporting；报告会进入私密安全通报流程，仅对报告者和被授权维护者可见。

不要为安全问题创建公开 Issue、讨论或 Pull Request，也不要把凭据、令牌、真实邮件或可利用细节发送到公开渠道。当前项目没有另行声明安全邮箱；请使用上面的 GitHub 私密通报入口，而不是猜测或查找未公开的联系方式。

一份有用的报告通常包括：

- 受影响的 Nami Mail 版本、Windows 版本和运行方式（桌面安装版或开发服务）；
- 安全影响和攻击前提，而不是只给出异常堆栈；
- 不含真实邮件、密码、令牌或个人地址的最小复现步骤；
- 可选的修复建议或安全测试。

如果必须分享敏感证据，请先在私有通报中询问安全传递方式。不要用 Base64、打码截图或压缩包替代访问控制。

## 常见安全边界

当前安全相关范围包括：

- Windows 桌面版的 DPAPI 主密钥保护、应用层 AES-256-GCM 加密和数据迁移；
- 本地回环 API、Electron preload/IPC、单实例和缓存策略；
- IMAP/SMTP TLS、账户认证、OAuth 回环回调和令牌保存；
- 邮件 HTML 清理、附件下载和本地文件处理；
- GitHub Release 更新源、ZIP/清单完整性、Authenticode 或 Ed25519 信任根、安装包验证和更新安装前的安全关闭。

以下情况通常不属于本项目可独立修复的漏洞范围，但仍可先私下讨论：邮箱服务商账户已失陷、管理员或同一 Windows 用户下的恶意进程、操作系统或 Electron 上游漏洞、用户主动安装的未签名第三方构建，以及服务商自身的 IMAP/SMTP/OAuth 可用性。

本项目的本地加密不是整库加密，也不能防御已解锁应用、管理员或同一 Windows 用户下能够调用 DPAPI 的恶意进程。详见 [隐私与本地数据说明](docs/PRIVACY.md)。

## 协调披露

请在修复发布前不要公开利用细节。维护者会确认影响、复现、修复和回归验证；在双方同意后，再通过 Release Notes、变更日志或安全公告说明受影响版本、缓解措施和致谢。没有复现、没有修复或仍可能危及用户时，不会承诺披露日期。
