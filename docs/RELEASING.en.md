# Windows Release Guide

[简体中文](RELEASING.md) | [English](RELEASING.en.md)

This guide is for maintainers. It describes Nami Mail's Windows installer, GitHub Release, and ZIP automatic-update flow. It describes only behavior that is implemented in code. A local build, unit tests, or a draft Release cannot replace a real installation test from an earlier version to a newer version.

## Release Model

Nami Mail desktop updates are available only from public GitHub Releases. The client carries no GitHub access token. It queries the public stable release at `https://api.github.com/repos/<owner>/<repo>/releases/latest`, then downloads a versioned JSON manifest and ZIP from that same public Release.

Every stable Release must contain exactly these five assets:

| Asset | Purpose |
| --- | --- |
| `Nami Mail Setup <version>.exe` | Windows NSIS installer. |
| `Nami Mail Setup <version>.exe.blockmap` | Installer block map generated by electron-builder. |
| `latest.yml` | Release metadata generated by electron-builder. Desktop ZIP updates do not use it as a source of trust. |
| `nami-mail-update-<version>-win-x64.zip` | ZIP update package containing exactly one root-level NSIS installer. |
| `nami-mail-update-<version>-win-x64.json` | Manifest with the ZIP version, filename, size, SHA-512, and optional Ed25519 signature. |

The release script first creates a draft Release, then downloads the five assets back from GitHub and compares their size and SHA-256 one by one. The draft is promoted to a stable Release only when asset names, count, and hashes all match. Do not manually upload, replace, or delete an asset and continue to regard that Release as verified.

## Prerequisites and Trust Roots

- Use Windows x64 and Node.js 22.14.0 or later.
- Use the public repository `QinIndexCode/nami-mail`, or update all build and release configuration if the repository moves.
- The `package.json` version must be an exact stable semantic version `x.y.z`; the tag must be exactly `v<version>`.
- The release output directory must be an isolated repository-relative directory, for example `release-artifacts/0.1.0`. The default output uses `release-artifacts/<package.json version>` for the current version. A production release and parallel verification must still set a directory explicitly and must not reuse old build output.
- Configure at least one update trust root. Production releases should use both timestamped Authenticode signing and Ed25519 manifest signing.

| Trust root | Required values | Runtime behavior |
| --- | --- | --- |
| Authenticode | `CSC_LINK` or `CSC_NAME`, plus `CSC_KEY_PASSWORD`, `NAMI_MAIL_EXPECTED_WINDOWS_PUBLISHER`, and `NAMI_MAIL_EXPECTED_WINDOWS_CERTIFICATE_THUMBPRINT` | If the currently installed program has a valid signature, the installer helper accepts only a new installer signed by the same publisher. |
| Ed25519 | `NAMI_MAIL_UPDATE_ED25519_PRIVATE_KEY`, a Base64-encoded PKCS#8 Ed25519 private key | Packaging derives only the public key and writes it to app resources. The client requires a manifest signed by that private key. |

If the current installer has no usable Authenticode signature, the release must include an Ed25519 public key and the Release manifest must have a valid signature. Otherwise, updates remain disabled. Private keys, certificates, certificate passwords, and `GH_TOKEN` may exist only in the current process or GitHub Secrets. Do not write them to `.env`, app resources, logs, commits, Release Notes, or issues.

GitHub Actions uses these Secrets:

| Secret | Required | Purpose |
| --- | --- | --- |
| `NAMI_MAIL_UPDATE_ED25519_PRIVATE_KEY` | Required without an Authenticode certificate | Base64 PKCS#8 Ed25519 private key; passed only to the packaging step of the release job. |
| `WINDOWS_CSC_LINK` | Required when using Authenticode | Certificate path, URL, or Base64 content usable by electron-builder. |
| `WINDOWS_CSC_KEY_PASSWORD` | As required by the certificate | Certificate password. |
| `WINDOWS_CSC_PUBLISHER` | Required when using Authenticode | Independently fixed expected signing publisher. |
| `WINDOWS_CSC_THUMBPRINT` | Required when using Authenticode | Independently fixed 40-character SHA-1 certificate thumbprint. |

GitHub Actions provides `GITHUB_TOKEN` temporarily to the release job. Do not create a long-lived token as a repository Secret. Public updates do not support embedding a private-repository token in the client.

## Local Preflight

Update the [changelog](../CHANGELOG.md) first and verify that user-visible copy, licensing, and the version number are correct. Run checks relevant to the release, then the complete baseline:

```powershell
npm.cmd ci
npm.cmd run build:brand:check
npm.cmd run typecheck
npm.cmd run test
npm.cmd --workspace @nami/web run test
npm.cmd run test:desktop-security
npm.cmd run build
npm.cmd run smoke:runtime
npm.cmd audit --omit=dev
```

When upgrading the major version of `better-sqlite3`, also perform physical database-compatibility verification against an independently checked-out older stable release. The check keeps a WAL database connection open with the older Electron, then uses the current Node to read, write, run `integrity_check`, checkpoint, and reopen it before the older release reads it again. It does not make the older release a current project dependency and does not fake an old binary in CI:

```powershell
npm.cmd run verify:legacy-sqlite-compat -- --legacy-root <older-stable-release-checkout>
```

The supplied path must be an independent worktree for an older stable version. It must not point at the active worktree or real user data. The older `better-sqlite3` must first be built for the older Electron ABI: in that independent worktree, run `npm.cmd ci`, `node node_modules/electron/install.js`, and `npm.cmd run rebuild:electron`, then run the command above. The compatibility check only reads the older worktree. It never downloads, rebuilds, or writes to it, and it creates a test database containing no business data in the system temporary directory and removes it when complete.

An ordinary local installer has no GitHub update channel and can be used for basic installer validation:

```powershell
$env:NAMI_MAIL_RELEASE_DIRECTORY="release-artifacts/0.1.0"
npm.cmd run package:win
npm.cmd run smoke:installer
```

GitHub update-package preflight uses a public repository, an isolated directory, and at least one trust root. The example below uses Ed25519. Provide a real private key only in a controlled environment. Never put a placeholder or private key in any file:

```powershell
$env:NAMI_MAIL_GITHUB_REPOSITORY="QinIndexCode/nami-mail"
$env:NAMI_MAIL_RELEASE_DIRECTORY="release-artifacts/0.1.0"
$env:NAMI_MAIL_UPDATE_ED25519_PRIVATE_KEY="<Base64 PKCS#8 Ed25519 private key>"
npm.cmd run package:win:github
```

When using Authenticode, also set `CSC_LINK` (or `CSC_NAME`), `CSC_KEY_PASSWORD`, `NAMI_MAIL_EXPECTED_WINDOWS_PUBLISHER`, and `NAMI_MAIL_EXPECTED_WINDOWS_CERTIFICATE_THUMBPRINT`. `package:win:github` validates repository visibility, builds the five assets, and runs packaging smoke tests, but does not create a stable Release.

## GitHub Actions Release

`.github/workflows/release-windows.yml` runs when a `v*` tag is pushed. The release script also rejects a tag that does not equal the `package.json` version. The workflow has two jobs separated by permissions:

1. `validate` has read-only permissions. It runs release-policy validation, type checking, server/web/desktop tests, runtime smoke, and production-dependency auditing.
2. `release` is attached to GitHub's `release` Environment. After validation succeeds and designated release owners approve it, it receives Release write permission and GitHub Secrets. Do not remove this human approval or run the release script in an unprotected environment.
3. `release` sets `NAMI_MAIL_RELEASE_DIRECTORY=release-artifacts/<tag>` and uses GitHub Secrets for signing or manifest signing.
4. The packaging step builds Electron/NSIS, creates the ZIP manifest, and runs packaging smoke locally on the runner. The release script then creates one unique draft Release and uploads the installer, blockmap, `latest.yml`, ZIP, and JSON manifest together. If any Release already exists for that tag or multiple drafts exist, the workflow fails. A maintainer must inspect and clean up the residue; do not combine partial assets.
5. The promotion step parses the same isolated directory again, downloads each remote draft asset, compares size and SHA-256, and only then publishes the stable Release.

Manual local publishing is only for maintainers with release permissions. It also requires `GH_TOKEN`, `GITHUB_REF_NAME=v<package.json version>`, and the same GitHub/trust-root environment variables. Prefer pushing a signed tag and letting the workflow release it, rather than mixing build environments on a personal workstation.

## Release Notes and Public Information

Every stable Release should provide brief user-facing notes instead of copying build logs. At minimum, state:

- The version number, major user-visible changes, fixes, and known limitations.
- Whether real public-network upgrade acceptance from an earlier version to a newer version has been completed. If it has not, explicitly state that it is pending verification.
- The Authenticode signing status of that exact installer. Without a valid signature, state that Windows SmartScreen may show an unknown publisher warning. Do not imply that users should bypass a system warning.
- Data compatibility, when users need to sign in or authorize again, and any irreversible migration.
- That users should download only `Nami Mail Setup <version>.exe` from this Release for manual installation. The versioned ZIP and JSON are internal automatic-update assets and must not be manually extracted or run.

This repository keeps publicly reviewable release text in the [Release Notes directory](releases/README.en.md). The first release can use the [v0.1.0 notes](releases/v0.1.0.en.md) directly, and the current `0.1.1` release uses the [published notes](releases/v0.1.1.en.md). The retained [candidate template](releases/v0.1.1-candidate.en.md) is only a pre-release checklist. The published notes deliberately retain the safety wording that real online upgrade acceptance has not yet been completed; only a later release may describe it as complete after the real verification at the end of this page has been completed and evidence retained.

Release notes must not contain `GH_TOKEN`, certificates, private keys, OAuth credentials, real mail, test-account data, or local absolute paths. If a released asset, signature, version, or update manifest is inconsistent, revoke or mark the Release first instead of silently replacing a file.

## Client Update Path

After the app starts, it checks the public stable Release after a delay and checks periodically while running. It accepts only a stable `x.y.z` tag newer than the current version. It rejects drafts, prereleases, downgrades, missing assets, and assets with incorrect names.

1. It reads the tag and asset list from `releases/latest`, and requires exactly matching versioned ZIP and JSON filenames.
2. It reads a JSON manifest no larger than 16 KiB and verifies the version, installer filename, ZIP filename, size, and SHA-512. If the current version uses an Ed25519 trust root, it also verifies the manifest signature.
3. After the user chooses "Update this version", it downloads the ZIP to `updates/<version>/` in the user-data directory. It verifies size and SHA-512 both during and after download.
4. After the user chooses "Restart and update", the app safely stops the local mail service and send queue. A helper process verifies the ZIP again and rejects an archive containing multiple files, a path hierarchy, or an unexpected installer. With an Authenticode trust root, it also verifies the signer of the current program and extracted installer. The Ed25519 path verifies the manifest signature before download and uses the SHA-512 bound by that manifest to pin the extracted installer; it does not treat metadata other than the JSON or ZIP as a source of trust.
5. The NSIS installer runs as a silent upgrade. On success, the helper attempts to clean the ZIP and temporary extracted directory up to five times. After each failure, it waits an increasing 100 ms interval and checks whether the path still exists. If residue remains, it records a non-sensitive `cleanup` phase. On startup, the new version performs a limited best-effort cleanup only for the matching version's update cache. A successful cleanup shows a non-interrupting result; a remaining cleanup failure keeps an error state. If installation itself fails, the helper records version, phase, and time, makes a best effort to restart the old version after checking the pre-start hash, and shows a failure state when the user later opens the app if it cannot restart automatically. Other processes for the same Windows user can lock or rename files, so this is not an absolute deletion guarantee.

Discovering an update does not download or restart automatically. A user can update, skip the current version, or defer for one hour, until tomorrow, one week, or 30 days. Skipping deletes the downloaded ZIP for that version. Network, TLS, GitHub rate-limit, missing-asset, and integrity failures show different actionable error messages, and no failed path installs a file that has not passed verification.

## Real Post-Release Update Validation

Automatic updates count as verified only with a real public Release, an installed older version, a published newer version, and actual network conditions. In an isolated test-data directory, complete at least this acceptance process:

1. Build and install an older stable version, such as `0.1.0`, using the same trust root. Create a disposable account or demo data.
2. Push an exact tag for a newer version, such as `0.1.1`, confirm GitHub Actions publishes it as stable, and verify that the Release page contains exactly five assets.
3. Start the older version. Confirm it requests the public GitHub Release, shows the in-app prompt, and separately validate "Update this version", "Skip this version", and "Remind me later".
4. Choose update and complete the download. Confirm the cached ZIP size and SHA-512 are verified. Choose restart and update, then confirm the installer finishes, the old process exits, the new version starts, and accounts and local settings remain.
5. After a normal update, confirm the ZIP and temporary extraction directory under `updates/<version>/` are cleaned. If Windows file locks leave residue, confirm startup retries only that version directory and shows user-visible state instead of deleting other versions or arbitrary paths. Retain reviewable evidence separately for a network interruption, TLS failure, 404/missing asset, tampered manifest or ZIP, signing failure, sync that did not stop safely, and duplicate launch.

Development mode, an ordinary local installer, and an installed build with no GitHub update configuration or trust root must show updates as unavailable. They must not be recorded as a verified update channel. After release, record verified versions and known limitations in the [README](../README.en.md), [changelog](../CHANGELOG.md), and Release Notes.
