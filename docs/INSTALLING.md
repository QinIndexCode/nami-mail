# Windows 安装与更新指南

[简体中文](INSTALLING.md) | [English](INSTALLING.en.md)

本指南面向日常使用 Nami Mail 的 Windows 用户。Nami Mail 当前的桌面发行范围是 Windows x64；从源码启动的 Web 或开发桌面会话不等同于正式安装版，也不会作为自动更新验证对象。

## 从可信来源安装

1. 打开项目的 [GitHub Releases](https://github.com/QinIndexCode/nami-mail/releases) 页面，确认仓库地址为 `QinIndexCode/nami-mail`。
2. 下载与目标版本同名的 `Nami Mail Setup <version>.exe`。不要下载或手动运行 `nami-mail-update-*.zip`、`.json`、`latest.yml` 或 `.blockmap`；它们是发布校验或应用自动更新的内部资源。
3. 关闭正在运行的 Nami Mail，再运行安装程序。安装过程中可以选择安装目录。

早期或未签名的 Windows 安装程序可能被 SmartScreen 标为“未知发布者”。这不是文件可信的证明，也不是要求你绕过系统警告的理由。请先回到上面的官方 Release 页面核对仓库、版本和 Release Notes；来源、版本或签名状态不清楚时，停止安装并通过 [支持指南](../SUPPORT.md) 报告。

## 已安装版本的处理

- 安装同一版本时，安装程序会让你选择重新安装或保留现有安装；
- 安装较新版本时，会按升级处理并保留本地数据；
- 运行较旧安装程序时，会提示这是降级并默认取消。只有明确确认后才继续，避免意外把新数据交给旧版本；
- Nami Mail 只允许一个桌面实例。再次打开时会恢复并聚焦已运行的窗口，而不会额外启动一套邮件同步或本地数据库。

## 首次启动和添加邮箱

1. 从开始菜单或桌面快捷方式打开 Nami Mail。
2. 选择“添加邮箱”，输入完整邮箱地址，等待应用完成服务商识别或显示手动配置入口。
3. 按当前服务商提示选择安全登录、应用专用密码、客户端授权码或账户密码。一次性短信/邮件验证码不能作为邮件客户端凭据。
4. 连接完成后，等待首轮同步。企业、学校和自定义域邮箱可能受管理员策略限制。

Google 和 Microsoft 的 OAuth 登录依赖已配置的公共客户端 ID。按钮显示为不可用时，不代表你的账号密码错误：请检查部署者是否提供了对应 OAuth 配置，或按 [邮箱接入指南](EMAIL-PROVIDERS.md) 选择该服务商允许的认证方式。不要把 OAuth client secret、验证码或普通网页登录密码填入不匹配的输入框。

## 自动更新

只有已安装、已配置公开 GitHub Release 通道且拥有有效更新信任根的 Windows 发行版会检查更新。开发会话、普通本地构建或没有更新配置的安装版会明确显示更新不可用。

发现更高稳定版本后，应用会显示三种选择：

- **更新此版本**：在后台下载并校验 ZIP 更新包。下载完成后，仍由你选择何时“重启并更新”；不会因为发现更新而自行重启。
- **跳过此版本**：不再提示该版本，并清理该版本已下载的更新缓存。
- **稍后提醒**：可选择 1 小时、明天、一周或 30 天后再次提醒。

更新会先检查版本、资源名称、大小、SHA-512 和发行信任链；安装前还会安全关闭本地服务与发件队列。成功后会尽力删除更新 ZIP 和临时展开目录。Windows 上的文件锁定可能使清理延后到下次启动，这不会影响已经安装的邮件数据。更新失败时，应用应继续可用，并显示可操作的错误说明；不要从不明链接下载替代 ZIP。

## 本地数据、关闭与卸载

桌面安装版的数据保存在当前 Windows 用户的 `%APPDATA%\Nami Mail`，不在安装目录中。邮件敏感载荷、凭据、OAuth 刷新令牌、发件队列和出站附件使用应用层加密；Windows 主密钥使用当前用户的 DPAPI 保护。加密范围、明文元数据与威胁边界请阅读 [隐私与本地数据说明](PRIVACY.md)。

关闭窗口时，可按应用设置选择退出或最小化到托盘。最小化到托盘会继续同步；退出或更新会先正常关闭本地服务。

卸载时默认保留本地数据。选择“同时删除本地数据”只会删除当前 Windows 用户的 Nami Mail 数据目录，包括本地数据库、密钥和设置；不会删除邮箱服务商上的邮件。删除前请先备份你确实需要保留的数据，并理解 DPAPI 保护的数据与当前 Windows 用户上下文绑定。

## 需要帮助时

连接或 TLS 问题先检查网络、DNS、代理、安全软件、系统时间、服务器地址和加密方式；认证问题先确认 IMAP/SMTP 已开启，并重新生成服务商要求的应用专用密码或授权码。提交 Issue 时只提供脱敏的版本、运行方式、错误分类和最小复现步骤，绝不上传真实邮件、地址、附件、OAuth 参数、令牌或密码。详情见 [支持指南](../SUPPORT.md)。
